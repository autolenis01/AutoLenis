# Supabase Migration Runbook — AutoLenis Platform

## Overview

This document defines the **migration procedures, rollback plans, release gating criteria, and RLS deployment steps** for all Supabase and schema changes in the AutoLenis platform. Every migration must satisfy the requirements defined in `SUPABASE_MASTER_SPEC.md`.

> **See also:** `docs/AUTH_MIGRATION_SUPABASE_RLS_RUNBOOK.md` for existing RLS migration procedures and `docs/SCHEMA_MIGRATION_NOTES.md` for schema migration history.

---

## 1. Migration Principles

1. **Prisma is the schema source of truth** — all schema changes originate as Prisma migrations
2. **RLS policies are co-migrated** — every schema change that adds or modifies a user-facing table must include RLS policy updates
3. **Backward compatibility is mandatory** — migrations must not break existing application code until the corresponding application release is deployed
4. **Rollback plans are required** — every migration must have a documented rollback path
5. **Staging validation is required** — migrations must succeed in staging before production deployment

---

## 2. Migration Workflow

### 2.1 Standard Migration Flow

```
1. Modify Prisma schema
   │
   ▼
2. Generate migration
   pnpm prisma migrate dev --name <migration_name>
   │
   ▼
3. Write RLS policy migration (if applicable)
   supabase/migrations/<timestamp>_<name>.sql
   │
   ▼
4. Write seed/update scripts for enums and settings
   │
   ▼
5. Test locally
   pnpm prisma migrate deploy
   pnpm test:unit
   │
   ▼
6. Create PR with migration + RLS + seeds + tests
   │
   ▼
7. Staging deployment and validation
   │
   ▼
8. Production deployment
```

### 2.2 Migration File Naming

| File Type | Location | Convention |
|---|---|---|
| Prisma migration | `prisma/migrations/<timestamp>_<name>/migration.sql` | Auto-generated by Prisma |
| RLS policy | `supabase/migrations/<timestamp>_<name>.sql` | Manual, descriptive name |
| Seed script | `prisma/seed.ts` or `scripts/seed-<name>.ts` | Purpose-specific |
| Rollback script | `scripts/rollback-<timestamp>_<name>.sql` | Mirrors migration |

## 3. Pre-Migration Checklist

Before applying any migration:

- [ ] Prisma schema validates: `pnpm prisma validate`
- [ ] Migration SQL is reviewed for correctness
- [ ] RLS policies are co-authored for new/modified tables
- [ ] Backward compatibility is verified (no breaking changes to existing queries)
- [ ] Rollback SQL is written and tested locally
- [ ] Enum/settings seed scripts are updated if needed
- [ ] Unit tests cover new schema behaviors
- [ ] RLS tests cover new/modified policies

---

## 4. Deployment Procedure

### 4.1 Staging Deployment

```bash
# 1. Apply Prisma migrations
pnpm prisma migrate deploy

# 2. Apply RLS policy migrations
# Via Supabase CLI or direct SQL execution
supabase db push

# 3. Run seed scripts if applicable
pnpm prisma db seed

# 4. Validate migration
pnpm prisma validate
pnpm typecheck
pnpm test:unit

# 5. Run RLS verification tests
pnpm test:unit -- rls

# 6. Run integration tests
pnpm test:unit -- integration
```

### 4.2 Production Deployment

```bash
# 1. Verify staging migration is clean
# 2. Create a database backup (Supabase dashboard or pg_dump)

# 3. Apply Prisma migrations
pnpm prisma migrate deploy

# 4. Apply RLS policy migrations
supabase db push

# 5. Run seed scripts if applicable
pnpm prisma db seed

# 6. Verify application health
# - Check API endpoints respond correctly
# - Check auth flows work
# - Check payment webhooks process
# - Monitor error rates for 15 minutes
```

### 4.3 Maintenance Window Criteria

Migrations requiring a maintenance window:

| Change Type | Window Required | Reason |
|---|---|---|
| Add nullable column | No | Non-breaking |
| Add required column with default | No | Non-breaking |
| Add required column without default | **Yes** | Requires data backfill |
| Rename column | **Yes** | Breaks existing queries |
| Drop column | **Yes** | Breaks existing queries |
| Add index on large table | Conditional | May lock table depending on size |
| Modify enum values | No | Additive enums are safe |
| Remove enum value | **Yes** | May break existing data |
| Add RLS policy | No | Additive, non-breaking |
| Modify RLS policy | Conditional | May change access behavior |
| Remove RLS policy | **Yes** | May expose data |

## 5. Rollback Procedures

### 5.1 Rollback Decision Matrix

| Severity | Action | Timeline |
|---|---|---|
| No errors, staging passed | Continue monitoring | — |
| Minor errors, workaround exists | Fix-forward in next release | < 24 hours |
| Data integrity risk | Immediate rollback | < 1 hour |
| Security exposure | Immediate rollback + incident | < 30 minutes |

### 5.2 Rollback Steps

```bash
# 1. Identify the problematic migration
pnpm prisma migrate status

# 2. Apply rollback SQL
psql $DATABASE_URL < scripts/rollback-<timestamp>_<name>.sql

# 3. Rollback RLS policies if changed
psql $DATABASE_URL < scripts/rollback-rls-<timestamp>.sql

# 4. Verify rollback
pnpm prisma validate
pnpm test:unit

# 5. Deploy previous application version
# (via Vercel rollback or git revert + deploy)

# 6. Document incident
# - What failed
# - Root cause
# - Rollback steps taken
# - Time to recovery
```

### 5.3 Rollback Script Template

```sql
-- Rollback: <migration_name>
-- Date: <date>
-- Author: <author>
-- Description: <what this rollback undoes>

BEGIN;

-- Reverse schema changes
-- Example: DROP COLUMN, remove index, etc.

-- Reverse RLS policy changes
-- Example: DROP POLICY, recreate previous policy

-- Reverse enum changes
-- Example: Remove added enum values (if safe)

COMMIT;
```

## 6. RLS Migration Procedures

### 6.1 Adding RLS to a New Table

```sql
-- 1. Enable RLS
ALTER TABLE "NewTable" ENABLE ROW LEVEL SECURITY;
ALTER TABLE "NewTable" FORCE ROW LEVEL SECURITY;

-- 2. Create policies
CREATE POLICY "buyer_select_own" ON "NewTable"
  FOR SELECT USING (
    auth.uid() IS NOT NULL
    AND "workspaceId" = current_workspace_id()
    AND current_role() = 'BUYER'
    AND "buyerId" = current_buyer_id()
  );

CREATE POLICY "admin_all" ON "NewTable"
  FOR ALL USING (
    auth.uid() IS NOT NULL
    AND "workspaceId" = current_workspace_id()
    AND is_admin()
  );

-- 3. Verify no open access exists
-- Run: SELECT * FROM pg_policies WHERE tablename = 'NewTable';
```

### 6.2 Modifying an Existing RLS Policy

```sql
-- 1. Drop the old policy
DROP POLICY IF EXISTS "old_policy_name" ON "TableName";

-- 2. Create the updated policy
CREATE POLICY "new_policy_name" ON "TableName"
  FOR SELECT USING (
    -- Updated conditions
  );

-- 3. Verify the change
-- Run: SELECT * FROM pg_policies WHERE tablename = 'TableName';
```

### 6.3 RLS Testing

After every RLS change, verify:

```sql
-- Test as anonymous (should fail)
SET LOCAL role TO 'anon';
SELECT * FROM "TableName" LIMIT 1;
-- Expected: 0 rows or permission denied

-- Test as authenticated buyer (should see own records only)
SET LOCAL role TO 'authenticated';
SET LOCAL request.jwt.claims TO '{"user_id": "<buyer_uuid>", "workspace_id": "<ws_uuid>", "role": "BUYER"}';
SELECT * FROM "TableName";
-- Expected: only buyer's own records in the correct workspace

-- Test workspace isolation (should not see other workspace data)
SET LOCAL request.jwt.claims TO '{"user_id": "<buyer_uuid>", "workspace_id": "<other_ws_uuid>", "role": "BUYER"}';
SELECT * FROM "TableName";
-- Expected: 0 rows
```

## 7. Release Gating Checklist

A release touching Supabase or schema is **not production-ready** unless all of the following pass:

### 7.1 Schema Validation

- [ ] `pnpm prisma validate` passes
- [ ] `pnpm prisma generate` completes without errors
- [ ] `pnpm typecheck` passes

### 7.2 Migration Validation

- [ ] Migration applies cleanly in staging
- [ ] Rollback script exists and is tested
- [ ] Backward compatibility verified

### 7.3 RLS Validation

- [ ] RLS is enabled on all new/modified user-facing tables
- [ ] Buyer isolation tests pass
- [ ] Dealer isolation tests pass
- [ ] Affiliate isolation tests pass
- [ ] Admin scope tests pass
- [ ] Cross-workspace isolation tests pass
- [ ] Backend-only tables are not browser-accessible

### 7.4 Integration Validation

- [ ] Auth/session tests pass
- [ ] Payment webhook idempotency tests pass
- [ ] Auction lifecycle tests pass
- [ ] Affiliate commission tests pass
- [ ] Contract Shield gate tests pass
- [ ] Signed URL storage tests pass
- [ ] Admin audit log assertion tests pass

### 7.5 Operational Validation

- [ ] Database backup created before production migration
- [ ] Monitoring alerts configured for error rate spikes
- [ ] On-call engineer identified for post-deployment support
- [ ] Rollback procedure reviewed by deployment engineer

## 8. Emergency Migration Procedures

### 8.1 Hotfix Migration

For critical fixes requiring immediate schema changes:

1. Create migration directly on a hotfix branch
2. Apply to staging and verify within 30 minutes
3. Apply to production with on-call engineer monitoring
4. Backport to main branch via PR
5. Document in incident report

### 8.2 Data Repair

For data corruption or integrity issues:

1. Identify affected records with diagnostic queries
2. Create a backup of affected tables
3. Write and review repair SQL
4. Apply in a transaction with rollback capability
5. Verify data integrity post-repair
6. Document in incident report
